# AEO 审核系统产品需求文档 (PRD)

## 1. 项目概述
本系统旨在为企业提供 AEO（高级认证企业）合规审核与资料管理功能。核心目标是通过标准化的路径对应关系和自动化的数据流，确保物理文件管理、CSV 配置与数据库状态的高度统一。

## 2. 核心功能需求

### 2.1 目录结构与数据同步 (单项标准)
- **权威数据源**：以 `audit_list_3.csv` 为唯一事实来源。其 `脱敏文件路径` 必须与物理文件系统路径严格 1:1 对应。
- **“路径即真理”同步逻辑**：
  - 系统需支持根据 CSV 路径自动构建数据库文件夹树（`folders`）。
  - 自动维护闭包表（`folder_closure`），确保多级层级关系正确。
  - **同步自愈**：当 CSV 中出现新路径时，系统应自动创建相关节点，无需人工干预 SQL。
- **命名规范化**：文件夹统一遵循 `序号-名称` 格式（如 `21-1-定位系统车载终端管理制度`），去除冗余分类前缀。

### 2.2 企业业务主体性质过滤 (Customization)
- **业务场景**：企业在系统设置中可选“物流”、“仓储”、“报关”、“外贸”一种或多种性质。
- **动态绑定逻辑**：
  - 系统应根据企业的性质，自动筛选并展示相关的“单项标准”审核项目。
  - **进度计算**：项目的总数（分母）应随绑定项目的增减而动态调整，仅计算合规范围内的进度。

### 2.3 资料管理展示逻辑
- **全路径命名 (Simplified Concatenation)**：
  - 审核项目名称展示应包含完整的层级路径，格式为：`顶级大类名称 - 二级目录名称 - 审核项目文件夹名称`。
  - **原则**：后端仅执行简单的字符串拼接展示，不做任何正则表达式过滤或关键词剔除，确保“所见即所得”。
  - **维护**：名称的准确性由用户通过规范物理文件夹/数据库命名来直接控制。

## 3. 技术架构规范

### 3.1 数据库设计
- 使用 `folders` 存储节点。
- 使用 `folder_closure` 存储树形闭包关系。
- 使用 `folder_check_files` 作为审核项目与文件夹的桥接表。

### 3.2 性能与缓存
- 系统采用 Hyperf 框架，重度依赖缓存。
- **缓存策略**：任何涉及文件夹结构、ID 偏移或分类命名的数据库变更，必须执行 `redis-cli flushall` 以强制清除 JSON 缓存块，并重启后端容器。

## 4. AI 检测与审计规范

### 4.1 关键词检测逻辑 (Keyword Audit)
- **触发条件**：当审核项的 `检测方式` 设置为 `关键词检测` 时触发。
- **PDF 提取范围限制**：为了平衡性能与有效性，系统**仅提取 PDF 文件的前 2 页**作为判定依据。
- **判定标准**：文件内容中必须包含预设的 `检测关键字`。如包含多组关键字（以“、”或“或”分隔），满足其一即视为通过。

## 5. 审核流水线标准化 (V4 标准 - 2026-04-25)

### 5.1 Unified `check_type` 定义
为了确保前端展示、后端逻辑与 AI 判定引擎的绝对对齐，系统统一了 `check_type` 字段的数值定义：

| 数据库值 | 定义 | 业务描述 | 判定逻辑 |
|---|---|---|---|
| **1** | **AI 内容理解** | AI Semantic/Numerical | 调用专用 Handler (如 FinancialRatio) 进行数值提取或语义判定 |
| **2** | **关键词检测** | Keyword Match | 调用 KeywordMatchHandler 扫描文档是否包含特定业务词汇 |
| **3** | **上传即为通过** | Simple Pass | 文档上传后无需 AI 介入，系统自动标记为“合格”状态 |

### 5.2 权威数据源切换 (Cloud-to-Local)
针对生产环境手动调优频繁的场景，系统采用 **“云端管理，本地备份”** 模式：
1. **线上调整**：所有审核逻辑、关键词、检测类型优先在线上数据库调整并验证。
2. **反向同步**：通过 `sync_db_to_csv.py` 脚本将线上最新的 `check_type` 和 `check_text` 导出并覆盖本地 `audit_list_3.csv`。
3. **闭环持久化**：将更新后的 CSV 提交至代码库，作为新环境部署的权威依据。

## 6. 维护与更新流程 (V4 规范)
1. **云端导出**：在服务器运行 `mysqldump ... > cloud_audit_structure.sql`。
2. **本地回填**：运行 `python3 sync_db_to_csv.py` 更新本地 CSV。
3. **物理校验**：确保本地磁盘/OSS 路径与 CSV 中的 `full_path` 严格一致。
4. **代码提交**：提交 CSV 与 SQL 脚本变更。

## 7. AI 审核故障排查与诊断链路 (V4.1 - 2026-04-25)

为了解决 AI 审核结论模糊和匹配失效问题，系统建立了标准化的诊断链路。

### 7.1 AI Handler 匹配逻辑 (两步清洗法)
当一个审核任务进入队列时，系统通过 `CheckHandlerFactory` 进行处理器分配：
1. **路径剥离**：如果项目名包含 ` 、 `，仅取最后一节（如 `财务状况 、 审计报告` $\rightarrow$ `审计报告`）。
2. **前缀清洗**：使用正则 `^[\d.&-]+\s*` 剔除所有装饰性序号（如 `6-15-1 无犯罪记录...` $\rightarrow$ `无犯罪记录...`）。
3. **核心匹配**：将清洗后的核心名称与 `match` 列表对比。若匹配失败，则落入 `KeywordMatchHandler` 兜底。

### 7.2 结论判定协议 (Result Protocol)
AI 任务输出的 `result_str` 必须遵循以下反馈规范：

| 结论前缀 | 含义 | 故障排查建议 |
|---|---|---|
| **[ERROR]** | **技术性识别失败** | 检查图片清晰度、文件是否加密、或 AI Prompt 是否需要微调。 |
| **(数值/描述)** | **业务性不合格** | AI 已成功读取数据，但数据不达标（如负债率 98%）。属于业务风险。 |
| **〔未能读取...〕**| **资料缺失** | 资料库中找不到对应年份或类型的关键页面。需检查上传范围。 |

### 7.3 典型故障诊断步骤
1. **现象：提示“未读取到有效关键字信息”**
   - **原因**：原本应由 AI 语义分析的项目误匹配到了关键词 Handler。
   - **检查点**：核对 `CheckHandlerFactory.php` 中的核心名称是否与数据库 `check_name` 去皮后一致。
2. **现象：审核状态一直显示“失败”或结论为空**
   - **原因**：AI 返回了非标准 JSON 或任务进程崩溃。
   - **检查点**：查看后端 `storage/logs/hyperf.log`，搜索 `GetAiResult` 关键字获取捕获的 Throwable 堆栈。
3. **现象：结论不准确（如年份读错）**
   - **原因**：AI Prompt 泛化能力不足。
   - **检查点**：修改 `storage/prompts/` 目录下对应的 Markdown 提示词，增加更严格的约束。
