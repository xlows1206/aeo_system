# AEO System 数据流与状态追踪 (Data Pipeline)

## 全局流向与状态
- [ ] 前端请求 (Vue/Nginx) -> [ ] Nginx 反向代理层 -> [ ] 后端 Hyperf / Swoole 服务入口 (9501 端口) -> [ ] 容器拦截与业务逻辑处理 -> [ ] 数据库响应
**当前阻滞点**：后端 Hyperf / Swoole 服务入口 (9501 端口)。

## 环节数据分析与问题记录

### 1. 部署与守护进程管理层
- **节点描述**：利用宝塔 Supervisor 管理 php / swoole Hyperf 常驻进程。
- **曾遇问题**：TCP 9501 地址被占用 (Address already in use[98])，导致新起进程失败。缺少 vendor 依赖。
- **解决状态**：已清理旧进程，补充了 vendor。
- **曾遇问题**：TCP 9501 地址被占用，同时框架输出缓存报 `Permission Denied`
- **解决状态**：已清理旧进程，已修复终端 root 测试与 www 用户守护进程间的权限割裂（无警告启动成功）。

### 2. 外部访问网络层
- **节点描述**：外部浏览器经过公网 IP 访问网站（可能经过 Nginx）。
- **曾遇问题**：收到 `ERR_EMPTY_RESPONSE` 报错，没发回任何 HTTP Data。并且后端日志没有收到任何请求活动的记录。
- **解决状态**：已在消除 Permission Denied 之后恢复该层。此时网络与 HTTP 握手数据打通。

### 3. Nginx 代理与前端分离层
- **节点描述**：前端打包文件（Vue）站立在 80 端口（独立管理，路径 `/www/wwwroot/admin`）。
- **现遇问题**：在管理面板中为 Hyperf 服务错误创建了一个使用 9501 端口的 Nginx 代理站点，导致与 Swoole 底层严重冲突。
- **原因判定**：用户按照传统 PHP-FPM 框架的部署习惯，试图把 Hyperf 当作普通的 PHP 网站去创建。实际上，Hyperf(Swoole) 是一个完全独立的底层 Web 服务器，它自带 HTTP 处理引擎并会原生地占据 9501，绝对不需要（也不能）在宝塔被注册成“Nginx网站”。
### 4. 前端 Axios / Fetch 请求发起层
- **节点描述**：前端 Vue 代码在浏览器中发起网络请求。
- **现遇问题**：网页加载成功，但在请求登录跨域验证时报错 `405 Method Not Allowed` (OPTIONS preflight 请求)。
- **原因判定**：前端代码打包时，环境变量里的 API 基础路径配错了！它携带了 `:9501` 后缀（比如向 `http://IP:9501/api/...` 发请求）。这导致浏览器认为这是跨域请求，发起了 OPTIONS 查询，但 Hyperf 默认没有开启跨域支持。更重要的是，这**完美绕过了我们刚辛苦建好的 Nginx 80 端口代理桥梁**！
- **解决方案要求**：去前端 Vue 代码配置里，把线上环境的 `BASE_URL` 后面的 `:9501` 删掉，只保留类似 `/api` 即可。让所有请求全打向 80 端口，利用刚才配置的 Nginx 反代转发，从而完美规避跨域！

### 5. 业务逻辑与数据库层 (公司年份更新)
- **节点描述**：前端页面修改公司年份，通过 `PUT /api/v1/company/durationYear` 保存到后端。
- **现遇问题**：配置好跨域与 Nginx 代理后，前端调用接口报 `500 Internal Server Error`。
- **原因判定**：代码层面引用了 `$company->start_year` 和 `$company->end_year`，这是我们最近新加的业务需求字段。但在向云服务器（宝塔）部署上线时，只传了代码，没有在云服务器的 MySQL 数据库执行同步建表或新增字段的操作。导致 ORM 在执行 Save() 时因为找不到字段而崩溃报错 500。
- **解决方案要求**：在服务器终端中进入后端目录并执行迁移命令 `php bin/hyperf.php db:migrate`，将数据库结构与最新代码强制对齐。
